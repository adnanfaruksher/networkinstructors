OSPF LSA Types 1–7 Explained — With Real LSDB Output

Most CCNP candidates can name the LSA types. Far fewer can read a live LSDB and tell you exactly what each entry means. This article walks through all seven LSA types using actual output from a 5-router NIT Lab topology.

Why the LSDB Matters More Than the List

You can memorise "Type 1 is a Router LSA, Type 2 is a Network LSA" and still have no idea what you're looking at when you run show ip ospf database on a live router. The exam tests the list. Production work tests your ability to read the database.

This article uses a 5-router topology in NIT Lab with Area 0 (backbone), Area 1 (standard), Area 2 (stub), Area 3 (NSSA), and an external redistribution source. Every LSA type is present and visible.

The Topology

Lab topology: R1 (ABR, Area 0↔1) — R2 (backbone, Area 0) — R3 (ABR, Area 0↔2 stub) — R4 (ABR, Area 0↔3 NSSA) — R5 (redistributes OSPF from a connected loopback, sits in Area 3)

Type 1 — Router LSA

Every OSPF router generates one Router LSA per area it belongs to. It describes the router's links within that area — interfaces, neighbors, and link costs. It floods only within its originating area.

R1# show ip ospf database router
            OSPF Router with ID (1.1.1.1) (Process ID 1)

                Router Link States (Area 0)

  LS age: 142
  Options: (No TOS-capability, DC)
  LS Type: Router Links
  Link State ID: 1.1.1.1
  Advertising Router: 1.1.1.1
  Number of Links: 2

    Link connected to: a Transit Network
     (Link ID) Designated Router address: 10.0.0.2
     (Link Data) Router Interface address: 10.0.0.1
      Number of MTID metrics: 0
       TOS 0 Metrics: 1

The key fields: Advertising Router is the RID of the originator. Link State ID for a Type 1 is also the originator's RID. Number of Links shows how many interfaces this router has in this area.

Type 2 — Network LSA

Generated by the DR on a multi-access segment (Ethernet). It represents the segment and lists all routers attached to it. Also floods only within its area. If there's no DR (point-to-point link), there's no Type 2 LSA.

R1# show ip ospf database network
                Net Link States (Area 0)

  LS age: 198
  LS Type: Network Links
  Link State ID: 10.0.0.2       <-- DR's interface IP
  Advertising Router: 2.2.2.2   <-- DR's RID
  Network Mask: /30
   Attached Router: 2.2.2.2
   Attached Router: 1.1.1.1

Type 3 — Summary LSA (Inter-Area)

Generated by ABRs to advertise networks from one area into another. When R1 (ABR between Area 0 and Area 1) sends Area 1 routes into Area 0, it generates Type 3 LSAs. These are the mechanism for inter-area route summarization.

R1# show ip ospf database summary
                Summary Net Link States (Area 0)

  LS age: 312
  LS Type: Summary Links(Network)
  Link State ID: 192.168.1.0   <-- network being advertised
  Advertising Router: 1.1.1.1  <-- the ABR
  Network Mask: /24
        MTID: 0         Metric: 1

Type 3 LSAs do not enter stub areas for external routes — they carry inter-area prefixes. This is why a stub area doesn't learn Type 5 externals but does learn inter-area routes via Type 3 (unless it's totally stubby).

Type 4 — ASBR Summary LSA

This tells routers in other areas how to reach the ASBR (the router doing redistribution). The ABR generates it when it sees a Type 1 LSA from an ASBR inside its area. Without Type 4, routers in Area 0 wouldn't know the path cost to reach R5.

R1# show ip ospf database asbr-summary
  LS Type: Summary Links(AS Boundary Router)
  Link State ID: 5.5.5.5       <-- ASBR's RID
  Advertising Router: 4.4.4.4  <-- ABR that generated it
  Network Mask: /0
        Metric: 10

Type 5 — AS External LSA

Generated by the ASBR for each redistributed external route. Floods throughout the entire OSPF domain — except stub and NSSA areas. This is why you can't redistribute into OSPF and have that route seen inside a stub area.

R1# show ip ospf database external
                Type-5 AS External Link States

  LS Type: AS External Link
  Link State ID: 172.16.0.0    <-- external prefix
  Advertising Router: 5.5.5.5  <-- ASBR's RID
  Network Mask: /16
        Metric Type: 2 (Larger than any link state path)
        Metric: 20
        Forward Address: 0.0.0.0

The metric type matters: E1 adds OSPF internal cost to the external metric; E2 (default) uses only the external metric regardless of internal cost. Most deployments use E2.

Type 7 — NSSA External LSA

Area 3 in our topology is an NSSA. When R5 redistributes into OSPF inside Area 3, it can't generate Type 5 (they're blocked at the area boundary). Instead it generates a Type 7 LSA that is local to the NSSA. The ABR (R4) then translates it to a Type 5 before flooding it into the backbone.

R4# show ip ospf database nssa-external
                Type-7 AS External Link States (Area 3)

  LS Type: AS External Link
  Link State ID: 10.99.0.0
  Advertising Router: 5.5.5.5
  Network Mask: /24
        Metric Type: 2
        Metric: 20
        Forward Address: 10.3.0.5    <-- next hop to ASBR
Key exam point: Type 6 (Multicast LSA) is defined but not implemented by Cisco. You will not see it in an LSDB on IOS. It exists in the spec only.

Quick Reference

LSA Type 1 — Router LSA — per-area, generated by every router, describes links within the area. Type 2 — Network LSA — per multi-access segment, generated by DR only. Type 3 — Summary LSA — inter-area routes, generated by ABR. Type 4 — ASBR Summary — path to ASBR, generated by ABR. Type 5 — AS External — redistributed routes, floods everywhere except stub/NSSA. Type 7 — NSSA External — redistributed routes inside NSSA only, translated to Type 5 by ABR.
